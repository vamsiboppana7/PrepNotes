1. What is the Facade Design Pattern?

The Facade Design Pattern is a structural pattern that provides a simplified interface to a complex system of classes, libraries, or frameworks.

GoF Definition: "Provide a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use."

2. Real-Time Example

In an e-commerce application, placing an order may involve:

  • Fetching product details
  • Processing payment
  • Sending invoice

Without the Facade pattern, the client would interact with each subsystem individually. With Facade, a unified interface (e.g., Order) simplifies this.

3. Class Diagram Components
  • Facade Class: Simplifies and unifies subsystem access
  • Subsystem Classes: Perform the actual work
  • Client: Uses the Facade instead of dealing with subsystems directly
4. Implementation in C#

Product.cs

public class Product {
            public void GetProductDetails() {
                Console.WriteLine("Fetching the Product Details");
            }
            }

Payment.cs

public class Payment {
            public void MakePayment() {
                Console.WriteLine("Payment Done Successfully");
            }
            }

Invoice.cs

public class Invoice {
            public void SendInvoice() {
                Console.WriteLine("Invoice Sent Successfully");
            }
            }

Order (Facade).cs

public class Order {
            public void PlaceOrder() {
                Console.WriteLine("Place Order Started");
                new Product().GetProductDetails();
                new Payment().MakePayment();
                new Invoice().SendInvoice();
                Console.WriteLine("Order Placed Successfully");
            }
            }

Client.cs

class Program {
            static void Main() {
                new Order().PlaceOrder();
                Console.Read();
            }
            }

Output:


            Place Order Started
            Fetching the Product Details
            Payment Done Successfully
            Invoice Sent Successfully
            Order Placed Successfully
5. Advantages
  • Simplifies access to complex systems
  • Decouples client from subsystems
  • Makes code easier to test and maintain
6. When to Use
  • When you need to simplify a complex set of classes
  • To provide an entry point into each layer of a layered system
  • To decouple clients from subsystem complexity

1. What is the Decorator Pattern?

The Decorator pattern is a structural design pattern that allows behavior to be added to individual objects, either statically or dynamically, without affecting the behavior of other objects from the same class. It acts as a wrapper to existing classes.

2. Key Concepts
  • Decorator class wraps the original class and provides additional functionality.
  • Maintains the class methods' signature intact.
  • Allows for dynamic behavior addition without modifying the original class.
3. Real-World Analogy

Consider a shape (like a circle or rectangle). You can add new features like color or border to it without altering the original shape class. This is achieved by wrapping the shape with a decorator that adds the new feature.

4. Implementation

We will create a Shape interface and concrete classes implementing this interface. Then, we'll create an abstract decorator class ShapeDecorator implementing the Shape interface and having a Shape object as its instance variable. RedShapeDecorator is a concrete class implementing ShapeDecorator. Finally, DecoratorPatternDemo will use RedShapeDecorator to decorate Shape objects.

Shape.java
public interface Shape {
                void draw();
            }
Circle.java
public class Circle implements Shape {
                @Override
                public void draw() {
                System.out.println("Shape: Circle");
                }
            }
Rectangle.java
public class Rectangle implements Shape {
                @Override
                public void draw() {
                System.out.println("Shape: Rectangle");
                }
            }
ShapeDecorator.java
public abstract class ShapeDecorator implements Shape {
                protected Shape decoratedShape;
            
                public ShapeDecorator(Shape decoratedShape){
                this.decoratedShape = decoratedShape;
                }
            
                public void draw(){
                decoratedShape.draw();
                }
            }
RedShapeDecorator.java
public class RedShapeDecorator extends ShapeDecorator {
            
                public RedShapeDecorator(Shape decoratedShape) {
                super(decoratedShape);        
                }
            
                @Override
                public void draw() {
                decoratedShape.draw();           
                setRedBorder(decoratedShape);
                }
            
                private void setRedBorder(Shape decoratedShape){
                System.out.println("Border Color: Red");
                }
            }
DecoratorPatternDemo.java
public class DecoratorPatternDemo {
                public static void main(String[] args) {
            
                Shape circle = new Circle();
            
                Shape redCircle = new RedShapeDecorator(new Circle());
            
                Shape redRectangle = new RedShapeDecorator(new Rectangle());
                System.out.println("Circle with normal border");
                circle.draw();
            
                System.out.println("\nCircle of red border");
                redCircle.draw();
            
                System.out.println("\nRectangle of red border");
                redRectangle.draw();
                }
            }
5. Output
Circle with normal border
            Shape: Circle
            
            Circle of red border
            Shape: Circle
            Border Color: Red
            
            Rectangle of red border
            Shape: Rectangle
            Border Color: Red
6. Conclusion

The Decorator pattern provides a flexible alternative to subclassing for extending functionality. It allows for dynamic behavior addition without modifying the original class structure, promoting code reusability and adherence to the Open/Closed Principle.
